<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| monitor → Etherscan API | Unauthenticated tx data from external API — fields must not be trusted for SQL without ? placeholders |
| monitor → Telegram send_message | user_id sourced from DB row (same row as wallet) — not from Telegram update context |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Denial of Service | _monitor_loop | mitigate | try/except CancelledError re-raise + catch Exception around _check_wallets; one crash doesn't kill loop |
| T-02-02 | Denial of Service | Etherscan rate limit in polling loop | mitigate | existing _throttle() in _get() enforces 0.25s between requests; applies automatically to get_transactions and get_current_block (INFRA-03) |
| T-02-03 | Tampering | update_last_block SQL | mitigate | UPDATE uses WHERE user_id=? AND address=? — prevents cross-user last_block updates |
| T-02-04 | Tampering | Etherscan tx data → DB insert | mitigate | mark_alert_sent uses INSERT OR IGNORE + UNIQUE constraint; tx_hash from Etherscan response not used in SQL string (? placeholder) |
| T-02-05 | Supply Chain | no new packages in this plan | accept | asyncio.create_task() used instead of APScheduler/job_queue — zero new dependencies |
</threat_model>
