<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Etherscan tx data → Telegram message | tx fields (hash, addresses, value_wei) appear in alert text — no HTML injection risk with Markdown mode but must not be used in SQL |
| monitor → Telegram API | alert sent to user_id from DB — correct user guaranteed by wallet ownership model |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-06 | Repudiation | Duplicate alerts | mitigate | mark_alert_sent called AFTER successful send; is_alert_sent checked BEFORE send (MON-04); UNIQUE(tx_hash, user_id) in DB |
| T-02-07 | Denial of Service | Failed send_message stops loop | mitigate | try/except per-tx wraps steps 4-6; one failed send logs error and continues to next tx |
| T-02-08 | Information Disclosure | Wrong user receives alert | mitigate | chat_id = wallet["user_id"] from DB row — same row that stores the user's own wallet; not from Telegram context |
| T-02-09 | Tampering | is_error=True tx triggers alert | mitigate | explicit if tx["is_error"]: continue before _should_alert check |
| T-02-SC | Tampering | npm/pip installs | accept | no new packages in this plan |
</threat_model>
